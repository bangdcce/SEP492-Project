UC ID and Name: UC-03 Login
Created By: SonNT Date Created: 2026-01-22
Primary Actor: User (any registered role) Secondary Actors: Authentication System
Trigger: The user opens the Sign In screen and selects "Sign In".
Description: Allow a registered user to sign in with email and password and enter the correct dashboard for their role.
Preconditions: PRE1: The user already has an account.
PRE2: The user has verified their email address (if email verification is required).
PRE3: The user has an internet connection.
Post-conditions: POS1: The user is signed in and can access protected features.
POS2: The user is redirected to the correct dashboard based on their role.
POS3: The system records the sign-in event for security.
Normal Flow: A. Sign In Successfully
  1. The user opens the Sign In screen.
  2. The user enters their email and password.
  3. The user selects "Sign In".
  4. The system validates the credentials.
  5. If valid and the account is eligible to sign in, the system signs the user in.
  6. The system redirects the user to the correct dashboard.
Alternative Flows: A3. Missing or invalid input
  1. The user leaves email/password empty or enters an invalid email format.
  2. The system displays a message asking the user to correct the input.
A5. Email not verified
  1. The system informs the user that email verification is required.
  2. The user checks their email and completes verification, then tries again.
A6. Wrong email or password
  1. The system informs the user that the credentials are incorrect.
  2. The user retries or selects "Forgot password".
A7. Connection problem
  1. The system cannot complete the sign-in request due to a connection/server issue.
  2. The system shows an error message and the user can retry.
Exceptions: E1: The system is temporarily unavailable and sign-in cannot be completed.
Priority: High
Frequency of Use: High
Business Rules: BR1: The user must provide valid credentials.
BR2: If email verification is required, unverified users cannot sign in.
BR3: The system records sign-in activity for security.
Other Information: If the user forgets the password, they can use the "Forgot password" option.
Assumptions: N/A


UC ID and Name: UC-04 Forgot Password
Created By: SonNT Date Created: 2026-01-22
Primary Actor: Guest (not signed in) Secondary Actors: Email Delivery System
Trigger: The user selects "Forgot password" from the Sign In screen.
Description: Allow the user to reset the account password by receiving a one-time verification code by email and setting a new password.
Preconditions: PRE1: The user knows the email address used for the account.
PRE2: The user can access that email inbox.
PRE3: The user has an internet connection.
Post-conditions: POS1: A verification code is sent to the user's email.
POS2: If the code is correct and not expired, the password is updated.
POS3: The user can sign in with the new password.
Normal Flow: A. Reset Password Successfully
  1. The user opens the "Forgot password" screen.
  2. The user enters their email and selects "Send code".
  3. The system sends a 6-digit code to the user's email.
  4. The user enters the code and selects "Verify".
  5. The system confirms the code is valid.
  6. The user enters a new password and confirms it.
  7. The user selects "Reset password".
  8. The system updates the password and shows success.
Alternative Flows: A2. Invalid email format
  1. The user enters an invalid email format.
  2. The system asks the user to correct the email.
A4. Resend code
  1. The user does not receive the email.
  2. After a short waiting time, the user selects "Resend code".
  3. The system sends a new code to the email.
A6. Wrong or expired code
  1. The user enters an incorrect or expired code.
  2. The system shows an error and allows retry or resend.
A8. Password does not meet requirements
  1. The new password is too weak or confirmation does not match.
  2. The system shows a message and asks the user to correct it.
Exceptions: E1: Email delivery is delayed or fails; the user can try again later.
Priority: High
Frequency of Use: Medium
Business Rules: BR1: The verification code is 6 digits and expires after a limited time.
BR2: The system may limit how often codes can be resent to prevent spam.
BR3: The new password must meet the platform's password rules.
Other Information: If the user no longer has access to the email account, they should contact support.
Assumptions: N/A

UC ID and Name: UC-07 Register KYC
Created By: SonNT Date Created: 2026-01-22
Primary Actor: Signed-in User (Client/Broker/Freelancer) Secondary Actors: KYC Review Team (Admin/Staff)
Trigger: The user starts identity verification (KYC) from the application.
Description: Allow the user to submit identity information and documents for verification so the platform can approve sensitive actions.
Preconditions: PRE1: The user is signed in.
PRE2: The user has required documents and photos ready (ID front/back, selfie).
PRE3: The user has an internet connection.
Post-conditions: POS1: The KYC submission is recorded in the system.
POS2: The KYC status becomes "Pending" until reviewed.
POS3: The user can see the current KYC status in the application.
Normal Flow: A. Submit KYC Successfully
  1. The user opens the KYC screen.
  2. The user enters personal information (name, date of birth, document number, address).
  3. The user uploads required images (ID front, ID back, selfie).
  4. The user confirms and submits.
  5. The system validates the submission and stores it.
  6. The system shows a confirmation message that the review is pending.
Alternative Flows: A3. Missing required information
  1. The user does not provide required fields or documents.
  2. The system highlights missing items and blocks submission.
A5. Invalid image file
  1. The user uploads a file that is not an image or exceeds the size limit.
  2. The system shows an error and asks the user to upload a valid image.
A7. Resubmission after rejection
  1. The user's previous KYC was rejected.
  2. The user corrects the information and submits again.
Exceptions: E1: Upload or save fails due to a system error; the user can retry later.
Priority: High
Frequency of Use: Medium
Business Rules: BR1: Only signed-in users can submit KYC.
BR2: KYC status follows a review flow: Pending, Approved, or Rejected.
BR3: Uploaded documents must be clear images and may be size-limited.
Other Information: The review process may take time. The user can continue using non-restricted features while waiting.

UC ID and Name: UC-08 Do Request Wizard
Created By: SonNT Date Created: 2026-01-22
Primary Actor: Client Secondary Actors: Wizard System
Trigger: The client starts creating a new request using the guided wizard.
Description: Guide the client through a step-by-step form to describe their project needs and choose how to proceed (save draft, post, or invite).
Preconditions: PRE1: The client is signed in.
PRE2: The client has enough information about the project to answer the wizard questions.
PRE3: Some actions may require approved KYC (depending on policy).
Post-conditions: POS1: The client completes the wizard steps and the system prepares a request summary.
POS2: Depending on the chosen action, the request is saved as a draft or submitted for visibility.
Normal Flow: A. Complete Wizard and Continue

  2. The system shows step-by-step questions (type, industry, budget/timeline, features, title/description).

    • Save Draft
    • Post to Market
    • Find and Invite
  5. The system confirms the action and shows a success message.

  1. The client skips a required step or leaves required fields empty.
  2. The system prevents moving forward until the information is provided.
A5. KYC is not approved (when required)
  1. The client tries to post or invite but KYC is not approved.
  2. The system informs the client to complete KYC or wait for approval.
  3. The client can still choose to save as draft.
Exceptions: E1: The system fails to save or submit due to a temporary error; the client can retry.
Priority: High
Frequency of Use: Medium
Business Rules: BR1: Only Clients can create requests using this wizard.
BR2: Required questions must be completed before submission.
BR3: Some actions may require approved KYC based on platform rules.
Other Information: A draft can be edited later before being posted.

Description: Allow a client to invite a broker to participate in a specific request.
Preconditions: PRE1: The client is signed in.
PRE2: The request exists and belongs to the client.
PRE3: The broker is available to be invited.
Post-conditions: POS1: The broker receives an invitation in their account/notifications.

Exceptions: E1: The system cannot send the invitation due to a temporary error; the client can retry.
Priority: High
Frequency of Use: Medium
Business Rules: BR1: A client can only invite brokers for their own requests.
BR2: The same broker should not be invited multiple times for the same request.
BR3: Invitations and responses are tracked by the system.
Other Information: Inviting a broker does not automatically hire them; the broker must respond.
